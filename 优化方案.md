# PrankLauncherBuilder 深度优化与演进方案 (Optimization Proposal)

基于当前 v3.0 版本的稳固架构（生成器界面、多线程并发扫描、代码签名与轻量混淆等），项目已具备较高的可用性。然而，为了进一步提升“恶搞成功率”、“隐蔽性”及“用户体验”，我们仍需针对现有的技术瓶颈进行深度优化。

---

## 一、 当前瓶颈分析

1. **引擎体积与特征**：
   - 当前的启动器 (Launcher) 基于 Python 和 PyInstaller 构建。即使不进行文件膨胀，单文件体积也通常在 10MB 以上。
   - PyInstaller 打包的 EXE 带有明显的启动解压特征和运行时指纹，容易被主流安全软件（如 360、火绒、Windows Defender）进行启发式拦截。
2. **免杀 (Evasion) 强度不足**：
   - 目前的代码混淆仅限于局部变量重命名和简单的字符串编码。缺乏控制流平坦化、虚拟机检测等高级对抗手段，容易被静态分析工具逆向还原逻辑。
3. **单向交互局限**：
   - 当启动器被目标受害者运行后，生成方（恶搞者）无法获得任何“战报”反馈。这降低了恶搞的成就感和闭环体验。
4. **GUI 表现力**：
   - 现有的 `tkinter` 界面虽然功能完善，但视觉效果相对传统，缺乏现代化的深色模式及动画交互。

---

## 二、 核心架构与免杀升级 (Core Engine & Evasion)

### 1. 启动器底层重构 (Rust / C++)
为了彻底摆脱 PyInstaller 的臃肿体积与固定特征，建议将 **Launcher 核心逻辑**（全盘扫描、启动、网页跳转）使用 **Rust** 或 **C/C++** 重写。
- **优化点**：
  - **体积缩减**：剥离 Python 解释器，目标体积从 >10MB 压缩至 <2MB。
  - **动态配置注入**：Builder 不再调用 PyInstaller，而是直接将配置数据注入到预编译的 C++/Rust 模板程序（PE 文件）的 Resource 段或通过 Overlay 附加，打包耗时将从十几秒缩减至毫秒级。

### 2. 深度环境对抗 (Anti-Analysis)
引入主动的防御机制，提高安全软件分析成本：
- **反沙箱/反虚拟机**：检测 CPU 核心数（<2 通常为沙箱）、MAC 地址厂商（VMware/VirtualBox）、物理内存容量。
- **反调试机制**：利用 `IsDebuggerPresent()`、`CheckRemoteDebuggerPresent` 等 Windows API，当发现自身处于调试环境中时，正常退出或展示良性行为（例如真实地打开一个记事本）。

---

## 三、 功能拓展与反馈闭环 (Features & Tracking)

### 3. Webhook “战报”回传机制
在 Builder 中增加网络回调配置：
- **优化点**：恶搞者可填入钉钉、飞书机器人或 Server酱 的 Webhook URL。
- **流程**：当受害者运行 Launcher 并在全盘扫描结束（成功打开目标程序或跳转至恶搞网页）时，后台异步发送一条 HTTP POST 请求：“🎉 目标已成功中招！当前状态：已拉起浏览器”。

### 4. 更多维度的 Payload (恶搞载荷)
除现有的“打开网页”外，增加更多模块化的恶搞行为供选择：
- **系统级伪装弹窗**：高仿 Windows 蓝屏界面或“系统关键文件丢失”的错误框。
- **音频惊吓**：打包时嵌入特定的 `.mp3` 或 `.wav` 文件（如大声尖叫或搞怪语音），在启动时隐蔽播放。
- **鼠标指针失控**：短时间内（如 5 秒）让鼠标光标轻微抖动或随机漂移。

---

## 四、 生成器体验升级 (Builder UX)

### 5. 现代化 UI 与所见即所得
- 引入 `customtkinter` 替代原生 `tkinter`，提供全平台一致的深色/浅色圆角 UI 风格。
- **实时预览**：在配置“伪装加载界面”的文案和背景图时，旁边提供一个迷你的预览窗（Preview Panel），即时展示修改后的效果。

### 6. 一键云端分发
- 集成公共的临时文件分享接口（如文叔叔、奶牛快传 API）。
- 打包成功后，提供“生成分享链接”按钮，自动上传生成的 EXE 并生成短链接/二维码，大幅降低用户的传输成本。
